# 關鍵字語音觸發流程

> 本文件說明 AI 智慧眼鏡系統中，語音喚醒詞「哈囉」的觸發流程。
> 包含喚醒詞模式（ESP32 眼鏡 / 本機 / APP 開啟喚醒詞）與 APP 旁路模式兩種路徑。
>
> 更新：2026-05-22 — 喚醒詞由「哈囉曼波」改為「哈囉」，結束詞已移除，新增音量鍵手動喚醒。

---

## 一、喚醒詞模式（ESP32 眼鏡 / 本機 / APP 開啟喚醒詞）

### 1.1 收音流程

```
使用者說話 → 麥克風收音
    → WebSocket /ws_audio → send_audio_frame()
    → AGC 自適應增益（音量太小放大、太大壓縮）
    → 放進 audio_queue
    → Google STT 串流辨識（cmn-Hant-TW，熱詞 boost=20）
    → _handle_result() 判斷待機/主動模式
```

### 1.2 待機模式（standby）

```
Google STT 回傳 final 結果
  → _check_wake_word(text)
    → is_wake_word(text)
       → _normalize_cn(text)（簡→繁 + 小寫）＋ 原文小寫
       → 是否包含 WAKE_WORDS 任一變體（哈囉/哈嘍/哈羅/…/hello…）
    → 命中？
       ├─ 是 → 說話人驗證（若啟用）→ on_wake()（播「開始對話」）+ enter_active_mode()
       └─ 否 → 印 log「無喚醒詞，忽略」
```

### 1.3 主動模式（active）

```
Google STT 回傳結果（partial 即時顯示，final 派發）
  → _handle_active(text, is_final)
    → final 結果 → 直接當指令派發給 AI（on_event）
  → 靜音超過 SILENCE_SEC(2.5s) 或超時 ACTIVE_MAX_SEC(12s)
    → 自動切回 standby（播「結束收音」）
```

> 已無結束詞 — 結束對話完全靠靜音 / 超時自動結束。

### 1.4 音量鍵手動喚醒（APP 限定）

```
APP 前景時，同時按「音量＋」與「音量－」
  → MainActivity.dispatchKeyEvent 偵測雙鍵同按
  → MethodChannel onManualWake → app_provider → websocket_service.sendAudioWake()
  → 送文字指令 "WAKE" 給 /ws_audio
  → server：非旁路模式 → recognition.enter_active_mode() + 播「開始對話」
            旁路模式   → 無害 no-op（麥克風本就全程收音）
```

## 二、APP 旁路模式（手機直連，預設）

```
APP 麥克風收音 → WebSocket /ws_audio，連線時送 START:BYPASS
  → bypass_wake = true
  → _check_wake_word() 直接跳過喚醒詞判斷
  → STT 結果直接派發給 AI（on_event）
  → 使用者不需要說「哈囉」，直接說指令即可
```

> APP 設定頁「喚醒詞」切換鈕：開 → 走第一節喚醒詞模式（送 `START`）；關 → 旁路模式（送 `START:BYPASS`，預設）。
> 硬體（ESP32 / 本機模式）一律走 `START`（非旁路），強制需要喚醒詞。

## 三、觸發 vs 不觸發範例（喚醒詞模式）

| 使用者說的 | 結果 | 原因 |
|-----------|------|------|
| 哈囉 | ✅ 觸發 | 含「哈囉」 |
| 哈囉幫我看前面 | ✅ 觸發 | 含「哈囉」（包含即命中）|
| 哈嘍 / 哈羅 / 哈摟 | ✅ 觸發 | 諧音變體 |
| 哈喽（簡體）| ✅ 觸發 | 簡體變體 |
| hello | ✅ 觸發 | 英文變體 |
| 你好 | ❌ 不觸發 | 不含「哈囉」變體 |
| 曼波 | ❌ 不觸發 | 舊喚醒詞已停用 |

> 注意：「哈囉」是日常高頻詞，誤觸發率較高 — 為專題展示「好呼叫」的刻意取捨。

## 四、Google STT 熱詞加成

`speech_contexts` 設定 `boost=20.0`，提高以下詞彙的辨識權重：

```
哈囉、哈嘍、哈羅、哈摟、哈漏、蛤囉、hello
```

目的是降低 ASR 誤辨機率；實際觸發判斷由 `is_wake_word()` / `WAKE_WORDS` 控制。

## 五、關鍵參數對照表

| 參數 | 值 | 說明 | 位置 |
|------|-----|------|------|
| `WAKE_WORDS` | 哈囉繁/簡/英變體集合 | 喚醒關鍵字（可由 `.env` 覆寫）| `asr_core.py` |
| `is_wake_word()` | — | 喚醒比對函式（包含即命中）| `asr_core.py` |
| `SILENCE_SEC` | `2.5` | 主動模式靜音超時（秒）| `asr_core.py` |
| `ACTIVE_MAX_SEC` | `12.0` | 主動模式最長錄音（秒）| `asr_core.py` |
| `speech_contexts boost` | `20.0` | 熱詞權重 | `asr_core.py` |
| `language_code` | `cmn-Hant-TW` | STT 語言（繁體中文）| `asr_core.py` |
| `WAKE`（`/ws_audio` 指令）| — | APP 音量鍵手動喚醒 | `app_main.py` |

## 六、設計決策記錄

### 為什麼喚醒詞是「哈囉」？

使用者指示：專題展示優先，「好呼叫」比抗誤觸發重要。已知並接受「哈囉」是日常高頻詞、誤觸發率高。

### 為什麼移除結束詞？

「謝謝」類詞太常出現於正常對話，當結束詞會誤截斷指令。改為純靠靜音 2.5 秒 / 超時 12 秒自動結束，更單純可靠。

### 為什麼音量鍵組合是「音量＋與音量－」而非含關機鍵？

Android 一般 App 無法攔截關機鍵（系統保留），且「音量－＋關機」是系統截圖手勢、無法覆蓋。故改用純音量雙鍵組合，且僅在 APP 前景且取得焦點時可攔截。
